# Project Guidance

- The current scaffold is a dependency-free Node.js ESM app. Keep new foundation work in plain JavaScript modules unless a later story explicitly introduces a framework or package.
- HTTP routing is centralized in `src/app.js`; `src/server.js` should stay limited to server startup concerns.
- Server-rendered HTML belongs in `src/pages.js`, shared product metadata belongs in `src/lib/product.js`, and mobile-first styling belongs in `public/styles.css`.
- Student-facing pages should render through the shared student shell helpers in `src/pages.js`; keep the four-tab bottom navigation data in `src/lib/product.js`, use page-type top bars, and hide bottom navigation only for task pages that retain a clear back entry.
- `npm run build` writes production files to `dist/src` and `dist/public`; `npm start` runs `dist/src/server.js`.
- Quality scripts in `scripts/` are intentionally dependency-free and should ignore Ralph run logs and PRD source files.
- MVP-wide quality gates include `src/e2e.test.js`, `scripts/data-quality.mjs`, and `scripts/browser-core-pages.mjs`; run `npm test`, `npm run data-quality`, and `npm run browser-test` before marking end-to-end UI quality complete.
- PostgreSQL schema migrations live in `src/db/migrations`; keep migration validation dependency-free until a later story introduces database tooling.
- Dependency-free seed fixtures and student-facing published-only data access helpers live in `src/db`; keep them in plain JavaScript until database tooling is introduced.
- Authentication/session helpers live in `src/auth.js`; route tests can inject an auth service through `handleRequest(request, response, { authService })`, and public responses should use the public user shape so phone fields never leak.
- HTML login-required student actions should use the shared `/login` flow with safe `returnTo` and bounded pending-action data; API-style requests should continue returning JSON `login_required` responses.
- Public school browsing cards should use `listSchoolGuideCards` from `src/db/data-access.js` so API and page behavior share published-only visibility, filters, sort order, timeline nodes, formula availability, and experience signals.
- Public school detail pages and APIs should use `getSchoolDetail` from `src/db/data-access.js` so year defaulting, published-only guide visibility, timeline/formula aggregation, and featured experience ranking stay consistent.
- Public school keyword matching includes school abbreviations from the seed/public JSON school shape; keep abbreviation search aligned across `listSchoolGuideCards`, guide keyword search, and rendered school cards.
- School detail HTML may label and show a latest-published historical fallback when a current/requested year is unpublished, but public school detail APIs should continue returning published-only 404 responses for unpublished requested years.
- School list filter enhancements in `public/student.js` replace server-rendered filter/results sections; preserve no-JS GET behavior and `data-school-filters-container`/`data-school-results-section` markers when changing this page.
- Public admission guide APIs should use `listGuides` and `getGuideDetail` from `src/db/data-access.js` so published-only current lists, keyword filters, source attribution, structured fields, and version summaries stay consistent.
- Admin official guide review should use the admin guide helpers in `src/db/data-access.js`; keep public guide APIs on `listGuides` and `getGuideDetail` so draft or pending_review guide records become student-visible only after an audited publish transition.
- Admin timeline and formula management should use `listAdminTimelineNodes`, `overrideAdminTimelineNode`, `listAdminFormulas`, `upsertAdminFormulaDraft`, and `publishAdminFormula` from `src/db/data-access.js`; keep public timeline and calculator APIs on published-only helpers so overrides and formulas become student-visible only after audited override/publication rules pass.
- Admin AI ingestion should use `createAdminIngestionRun`, `listAdminIngestionRuns`, and `getAdminIngestionRunDetail` from `src/db/data-access.js`; ingestion output can create draft guide records only, source candidates must preserve priority and traceability, and public guide APIs must stay published-only.
- Public timeline pages and APIs should use `listTimelineNodes` and `buildSiteTimelineReminders` from `src/db/data-access.js` so generated required nodes, status calculation, unknown-date handling, published-only visibility, and site-only reminder indicators stay consistent.
- Public timeline UI filters should expose student-facing `nodeType` in URLs/forms and map it to the data helper `eventKey` filter at the route boundary; unknown-date nodes can render as "to be announced" in HTML without changing the API status contract.
- Public score calculation should use `calculateScore` from `src/db/data-access.js` so formula visibility, input normalization, validation errors, breakdowns, source links, and disclaimer text stay consistent across APIs and pages.
- Public score calculator disclaimer copy comes from `calculateScore`; preserve the "not an admission probability" warning without adding ranking prediction, recommended application decisions, guaranteed admission, or similar anxiety-inducing language.
- Public experience browsing pages and APIs should use `listExperiences` from `src/db/data-access.js` so published-only visibility, filters, PRD default ordering, keyword matching, and sort order stay consistent.
- Public experience detail pages and APIs should use `getExperienceById` and `renderExperienceDetailPage`; keep `/experiences/:id` and `/api/experiences/:id` published-only and free of verification materials, phone fields, real names, source accounts, and private user ids.
- Structured experience submissions should go through `src/experience-submissions.js`; route tests can inject an `experienceSubmissionStore`, and public or anonymous submission output must not expose verification materials, source accounts, phone fields, or real names.
- Experience submission drafts live in browser localStorage under `gce:experience-submission-draft` for up to seven days; `public/student.js` should clear them on successful submission and logout, and hidden prompt/toast state depends on the global `[hidden]` CSS rule.
- Student interaction state for favorites, useful votes, and reports lives in `src/interactions.js`; route tests can inject an `interactionStore`, and public report or interaction outputs should not expose phone fields or private reporter data.
- Personal center routes should build user-owned aggregates in `src/app.js` from the public user shape, `interactionStore`, `experienceSubmissionStore`, site-only timeline reminders, and submission review-status reminders; HTML `/me` renders a logged-out My guide for visitors, while API-style personal endpoints still return JSON `login_required` responses.
- Admin experience moderation should use `src/experience-submissions.js` review helpers; approved submissions become student-visible through `registerPublishedExperience`, verification review should expose only reviewer-safe material metadata, and report resolutions should combine `interactionStore` audit state with public experience hiding/deletion or auth account-status updates.
