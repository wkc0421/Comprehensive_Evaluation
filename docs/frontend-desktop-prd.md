# 广东综评 MVP 电脑 Web 版前端 PRD

版本：v0.1
日期：2026-06-06
关联文档：`docs/prd.md`、`docs/frontend-prd.md`
产品形态：学生端电脑 Web + 管理端电脑 Web
设计基准：可信、清爽、适合横向扫描的信息工具站

## 1. 文档目标

本文件补充广东综评 MVP 的电脑 Web 版前端要求，覆盖学生端电脑 Web 和管理端电脑 Web。电脑端不改变 `docs/prd.md` 的产品范围、数据规则、权限规则和 API 逻辑，也不另起一套独立业务流程；它基于现有移动 Web 前端逻辑和管理端桌面后台逻辑，将信息展示、筛选、详情阅读、计算、投稿和审核流程适配到更宽的桌面视口。

学生端电脑 Web 重点解决三类体验：

1. 学生和家长在电脑上横向比较学校、年份、状态和关键时间。
2. 在院校详情页同时查看官方资料、时间线、公式和面经，不频繁上下跳转。
3. 在综合分计算、面经投稿等长表单场景中减少滚动和重复确认。

管理端电脑 Web 重点解决四类体验：

1. 运营人员以队列方式处理 AI 入库、简章审核、公式配置和时间线覆盖。
2. 审核人员在同屏查看结构化字段、官方来源、风险提示和学生端预览。
3. 管理员处理面经、认证材料和举报时，清楚区分后台可见材料和学生端可见内容。
4. 所有发布、退回、隐藏、删除、限制账号等动作有明确原因、状态和审计记录。

## 2. 产品边界

电脑端必须与现有移动端保持同一业务边界：

- 不做录取概率预测、排名预测、志愿填报决策建议。
- 不做学校热榜、分数榜、开放评论区、私信。
- 不展示或售卖未公开考试原题。
- 不自动发布 AI 抽取结果。
- 官方信息必须展示年份、来源、状态或待补充说明。
- 面经公共展示不得暴露手机号、真实姓名、认证材料、来源账号、私有用户 ID。
- 未登录用户可以查院校、看简章、看时间线、看计算器说明和浏览公开面经。
- 登录用户可以收藏、点有用、举报、投稿、查看个人收藏和投稿审核状态。

电脑端可以增加桌面布局、批量筛选、固定目录、右侧信息面板和更高密度列表，但不得增加新的业务判断或绕过移动端已有的登录、审核、发布和隐私规则。

手机版 Web 的管理端边界：

- `docs/frontend-prd.md` 已包含管理端页面设计，但管理端明确采用桌面优先布局。
- MVP 不单独设计手机版管理端；手机浏览器访问管理端时可以展示登录、无权限、基础错误页或提示使用电脑处理。
- 管理端核心审核、发布、认证材料查看、举报处理等工作流以电脑 Web 为验收目标。

## 3. 设计目标

### 3.1 高效扫描

- 首页、院校列表、时间线和面经列表优先支持横向扫描。
- 同屏可展示更多字段，但每个字段必须可读、可比较、可定位。
- 列表页支持左侧筛选区和右侧结果区，避免筛选项全部塞入弹层。

### 3.2 稳定可信

- 官方简章、时间节点、公式必须在桌面端持续展示来源和年份。
- 历史参考、待公布、待补充、已截止等状态不得弱化。
- 关键操作按钮旁保留风险或前置条件说明，例如无公式时不出现可计算入口。

### 3.3 任务连续

- 院校详情页要支持从一屏内进入官方简章、报名入口、计算器、面经。
- 综合分计算器在桌面端使用“选择/说明/输入/结果”并排或分区布局，减少反复滚动。
- 投稿面经页使用分组表单和侧边提交摘要，帮助用户确认必填项与隐私设置。

### 3.4 与移动端一致

- 电脑端页面的内容优先级、状态含义、文案边界和登录行为与 `docs/frontend-prd.md` 一致。
- 电脑端 URL、查询参数和 API 契约应尽量复用移动端实现。
- 无 JavaScript 时仍保留服务端渲染核心查询、筛选和详情阅读能力。

### 3.5 管理端高密度审核

- 管理端以队列表格、详情面板、来源预览和审核操作栏组织工作流。
- 管理端允许比学生端更高的信息密度，但不得牺牲字段可读性、操作确认和隐私隔离。
- 管理端所有会影响学生端可见内容的操作必须能看到学生端预览。

## 4. 适配范围

### 4.1 目标视口

电脑端验收视口：

- 1024px 宽：小屏笔记本或平板横屏。
- 1280px 宽：主验收桌面宽度。
- 1440px 宽：常见办公显示器。
- 1920px 宽：宽屏显示器，内容不得被无限拉宽。

最低可用宽度：

- 768px 到 1023px 使用过渡布局：顶部导航保留，主内容由多栏收敛为单栏或双栏。
- 低于 768px 继续遵守 `docs/frontend-prd.md` 的移动端布局。

### 4.2 内容宽度

- 学生端页面主体最大宽度建议为 1180px 到 1280px，居中展示。
- 学生端列表页筛选栏宽度建议为 240px 到 280px。
- 学生端详情页右侧辅助栏宽度建议为 300px 到 360px。
- 学生端表单页主表单宽度建议为 680px 到 760px，右侧摘要栏宽度建议为 300px 到 340px。
- 管理端主体最大宽度可以放宽到 1440px 到 1600px，优先满足表格、来源预览和右侧审核面板。
- 管理端左侧导航宽度建议为 200px 到 240px，右侧详情或审核面板宽度建议为 380px 到 520px。
- 1920px 宽屏下不得把正文行宽拉到难以阅读，长文本区域最大行宽不超过 760px。

## 5. 视觉方向

电脑端沿用移动端蓝绿色工具感视觉，但信息密度更高、装饰更克制。

### 5.1 色彩

- 继续使用 `docs/frontend-prd.md` 的色彩 token。
- 页面背景使用浅灰，内容区以白底或无框分区为主。
- 主色只用于主操作、选中导航、关键链接和焦点态。
- 状态标签使用浅底深字，不使用大面积高饱和色块。
- 不使用大面积渐变、装饰性背景、营销式横幅。

### 5.2 排版

- 中文字体使用系统默认无衬线字体。
- 正文 14-16px，表格正文可使用 14px，但不得低于 13px。
- 页面主标题 24-28px，列表区标题 18-20px，卡片标题 16-18px。
- 字间距保持默认，不使用负字距。
- 桌面端可以展示更多辅助字段，但避免长段说明塞进表格单元格。

### 5.3 间距

- 页面上下留白比移动端更克制，主内容区纵向间距 16-24px。
- 卡片或面板内边距 16-20px。
- 表格行高不低于 48px。
- 按钮、输入框高度不低于 40px；关键操作和表单提交按钮不低于 44px。
- 卡片圆角仍不超过 8px。

### 5.4 图标

- 顶部导航和按钮可使用图标加文字。
- 表格行内高频操作可使用图标按钮，但必须有可访问名称，可额外提供 tooltip。
- 不使用复杂插画作为主视觉。

## 6. 全局框架

### 6.1 顶部导航

电脑端学生页面使用固定或粘性的顶部导航，替代移动端底部四栏导航。

导航内容：

1. 产品名：广东综评
2. 主导航：首页、院校、时间线、计算器、面经、我的
3. 全局搜索入口：搜索学校、简章、面经关键词
4. 年份快捷切换：2026、2025、2024
5. 登录状态：登录按钮或用户入口

规则：

- 当前导航项使用主色或下划线表达，不能只依赖颜色。
- 顶部导航高度建议 56-64px。
- 小屏桌面宽度不足时，全局搜索和年份快捷切换可收敛到二级工具条或按钮；主导航至少保留首页、院校、面经、我的，时间线和计算器可进入更多菜单。
- 未登录点击“我的”、收藏、投稿、有用、举报时进入统一登录流程，并保留 `returnTo`。

### 6.2 页面布局

桌面学生端使用以下三类布局：

1. 概览布局
   - 用于首页。
   - 主内容区展示核心入口、最近节点、最新简章、最新面经。
   - 右侧可展示年级准备提示和登录/收藏提示。

2. 列表布局
   - 用于院校、时间线、面经列表。
   - 左侧为筛选区，右侧为结果区。
   - 结果区可使用表格、紧凑卡片或分组时间线。

3. 详情布局
   - 用于院校详情、面经详情、计算器、我的页面。
   - 主内容展示核心信息。
   - 右侧展示目录、关键操作、来源摘要、个人状态或提交摘要。

### 6.3 面包屑与返回

- 详情页顶部展示轻量面包屑，例如“院校 / 中山大学 / 2026”。
- 从列表进入详情后，返回应保留原筛选条件。
- 任务页退出前如有未保存内容，必须提示确认。

### 6.4 页面状态

- 加载中：列表和详情使用骨架屏，避免大面积空白。
- 无结果：保留筛选条件，并提供清空筛选或查看全部入口。
- 错误：展示重试按钮和可理解的错误说明。
- 未登录：展示登录引导，但不得阻断公开信息浏览。

### 6.5 管理端全局框架

管理端电脑 Web 使用桌面后台框架：

1. 左侧导航
   - 数据入库
   - 简章审核
   - 时间线管理
   - 公式管理
   - 面经审核
   - 认证审核
   - 举报处理

2. 顶部状态栏
   - 当前页面标题。
   - 当前登录管理员。
   - 任务状态或队列数量。
   - 退出登录入口。

3. 主内容区
   - 队列表格。
   - 筛选和批量定位工具。
   - 空状态、加载状态和错误状态。

4. 详情区
   - 右侧详情面板或抽屉。
   - 官方来源、结构化字段、风险提示和学生端预览。
   - 审核操作栏。

管理端不使用学生端底部导航，不使用学生端居中窄容器。

## 7. 学生端页面设计

### 7.1 首页

页面目标：

- 让电脑端用户快速进入四类核心任务。
- 提供最近节点、已发布简章和高质量面经的横向概览。
- 给不同年级用户稳定的准备提示。

首屏结构：

1. 顶部欢迎区
   - 标题：广东综评
   - 副标题：查简章、看节点、算综合分、读面经
   - 年级切换：高一、高二、高三

2. 核心任务入口
   - 查院校
   - 看时间线
   - 算综合分
   - 看面经

3. 最近节点面板
   - 展示 3-5 个最近节点。
   - 登录且有收藏时优先显示收藏学校节点。
   - 未登录或无收藏时显示全站最近节点。

首屏下方：

4. 最新简章
   - 桌面端可以使用紧凑表格或列表。
   - 字段：学校、年份、简章状态、报名截止、来源状态、入口。

5. 最新面经
   - 展示学校、年份、阶段、校测形式、认证标签、有用数。

6. 年级准备提示
   - 高一：了解综评、关注选科、建立材料意识。
   - 高二：核对学考和选科、关注目标学校变化。
   - 高三：盯紧简章、截止时间、校测准备。

桌面规则：

- 首页不能做营销落地页或门户式资讯首页。
- 同屏最多一个主按钮。
- 右侧辅助区不得遮挡或替代核心任务入口。

### 7.2 院校列表页

页面目标：

- 支持电脑端按年份、状态、学校类型和关键词快速筛选。
- 让学生和家长能横向比较学校、报名状态、公式和面经情况。

页面结构：

1. 左侧筛选区
   - 年份：2026、2025、2024
   - 简章状态：全部、已发布、待发布、历史参考
   - 报名状态：全部、未开始、进行中、已截止、待公布
   - 学校类型
   - 是否有公式
   - 是否有面经
   - 清空筛选按钮

2. 顶部结果工具条
   - 搜索框：搜索学校名称或简称
   - 排序：报名截止时间、更新时间、学校名称
   - 结果数量

3. 结果列表
   - 推荐使用表格或紧凑卡片列表。
   - 字段：学校、年份、简章状态、报名状态、报名截止、公式类型、面经数量、收藏、详情入口。

交互：

- 筛选改变后结果区局部刷新，并保留 URL 查询参数。
- 无 JavaScript 时通过 GET 表单提交保留筛选能力。
- 点击学校名称或详情入口进入院校详情页。
- 收藏要求登录；登录后继续执行收藏并返回当前筛选结果。

状态：

- 筛选无结果：展示“没有符合条件的院校，试试切换年份或状态”，并提供清空筛选。
- 数据加载失败：保留筛选栏和上次结果位置，展示重试按钮。
- 长学校名称换行或截断，但 tooltip 或详情入口应能查看完整名称。

### 7.3 院校详情页

页面目标：

- 在电脑端一页聚合学校年份的官方简章、时间线、成绩公式、招生要求和面经。
- 支持横向阅读和快速跳转，减少长页面迷路。

页面结构：

1. 顶部学校摘要区
   - 学校名称、简称、所在地。
   - 年份切换。
   - 简章状态、报名状态。
   - 收藏按钮。

2. 主内容区
   - 关键时间。
   - 官方简章摘要。
   - 成绩公式。
   - 招生要求。
   - 校测与录取。
   - 费用与咨询。
   - 同校面经。

3. 右侧固定信息栏
   - 页面目录。
   - 关键操作：官方简章、报名入口、算综合分、发布面经。
   - 来源摘要：来源类型、更新时间、简章年份。
   - 历史参考或待补充提示。

操作优先级：

- 报名进行中时，报名入口是唯一主按钮。
- 报名不在进行中且公式已发布时，算综合分可以作为唯一主按钮。
- 发布面经、收藏、查看全部面经使用次级按钮或文本链接。
- 无明确公开公式时不得展示可执行计算入口。

交互：

- 年份切换后刷新当前学校对应年份数据。
- 页面目录点击后滚动到对应分区。
- 官方链接在新页面打开。
- 长文本默认展示摘要，可展开完整内容。
- 当请求年份未发布但有历史简章时，HTML 可展示最近已发布历史参考，并明确标注“历史参考”；公开 API 仍按 `docs/prd.md` 保持未发布请求 404。

### 7.4 时间线页

页面目标：

- 支持电脑端按年份、节点类型和收藏状态查看综评节点。
- 让高三用户快速定位最近截止和待公布事项。

页面结构：

1. 左侧筛选区
   - 年份。
   - 节点类型。
   - 节点状态。
   - 范围：全部节点、我的收藏。

2. 主时间线区
   - 按月份分组。
   - 每个节点展示学校、节点类型、日期、状态、来源年份。
   - 即将截止节点在组内靠前展示。

3. 右侧提醒区
   - 最近 14 天内节点。
   - 登录用户收藏学校节点优先。
   - 未登录展示登录后可查看我的时间线提示。

交互：

- 点击节点进入对应院校详情页。
- 未登录点击“我的收藏”进入登录引导。
- 未知日期展示“待公布”，不得改变 API 状态契约。

### 7.5 综合分计算器页

页面目标：

- 在电脑端让学生按公开公式快速完成输入、计算和结果核对。
- 明确结果只是按公开公式换算，不代表录取概率。

页面结构：

1. 左侧选择与公式区
   - 学校选择。
   - 年份选择。
   - 公式状态。
   - 公式名称、权重说明、满分说明。
   - 官方依据链接。

2. 中间输入区
   - 按 `score_formulas.inputs_schema` 动态生成输入项。
   - 每个输入项展示名称、满分、是否必填、错误提示。
   - 数字输入框支持键盘输入。

3. 右侧结果区
   - 综合分。
   - 分项贡献。
   - 公式来源。
   - 免责声明：计算结果仅按公开公式换算，不代表录取概率。

交互：

- 学校或年份改变后清空已输入分数。
- 必填项未填时计算按钮禁用。
- 输入超出满分时在当前输入项旁提示。
- 计算后结果区更新，不输出建议报考、稳妥、危险、概率或排名。

状态：

- 无公式：展示“该校该年份暂无明确公开公式，暂不支持计算”，并隐藏输入和计算按钮。
- 公式草稿未发布：学生端不可见。
- 来源缺失：公式不得发布，学生端不可见。

### 7.6 面经列表页

页面目标：

- 支持电脑端按学校、年份、阶段、校测形式和认证状态查找面经。
- 保持结构化经验社区，不做开放评论和热榜。

页面结构：

1. 左侧筛选区
   - 学校。
   - 年份。
   - 阶段。
   - 校测形式。
   - 是否认证。
   - 排序：默认、最新、有用数。

2. 顶部搜索区
   - 搜索学校、阶段、关键词。
   - 发布面经入口。

3. 面经结果区
   - 可使用双列卡片或列表。
   - 字段：学校、年份、阶段、形式、摘要、认证标签、历史参考标签、有用数、收藏。

交互：

- 筛选结果局部刷新并保留 URL 查询参数。
- 点击卡片进入详情页。
- 发布面经要求登录；登录后进入投稿页并保留来源上下文。
- 审核中内容不展示在公共列表。

状态：

- 无结果：展示“还没有这类面经，可以换个筛选条件或发布第一条”。
- 超过 2 年的面经展示“历史参考”。
- 匿名面经不展示可识别投稿人信息。

### 7.7 面经详情页

页面目标：

- 让学生快速判断这条面经是否适合参考。
- 在电脑端清晰展示结构化经验和可信标签。

页面结构：

1. 主内容区
   - 头部信息：学校、年份、阶段、校测形式、认证标签、历史参考标签。
   - 基本信息：专业/招生大类、考生类型、是否入围、是否最终录取、考试地点。
   - 流程。
   - 问题类型。
   - 准备方式与建议。
   - 体验评分。

2. 右侧操作区
   - 收藏。
   - 有用。
   - 举报。
   - 同校面经入口。

规则：

- 不展示投稿账号手机号、真实姓名、来源账号、认证材料。
- 有用和举报要求登录。
- 举报成功后使用轻量反馈，不公开举报人信息。

### 7.8 发布面经页

页面目标：

- 降低电脑端长表单投稿成本。
- 帮助用户确认必填项、匿名展示和认证材料可见范围。

页面结构：

1. 主表单区
   - 基础信息：年份、学校、专业/招生大类、考生类型。
   - 经历结果：阶段、是否入围、是否最终录取、校测形式、考试地点。
   - 经验内容：大致流程、问题类型、准备方式、给下一届的建议。
   - 体验评分：难度、紧张程度、区分度。
   - 展示与认证：是否匿名、可选上传认证材料。
   - 发布须知。

2. 右侧提交摘要
   - 必填完成度。
   - 匿名展示状态。
   - 认证材料说明。
   - 提交按钮。

交互：

- 支持本地草稿，规则与 `docs/frontend-prd.md` 保持一致。
- 本地草稿只保存文字字段、选择项和评分，不保存认证材料文件。
- 认证材料只在用户主动提交时上传。
- 离开页面前如有未保存内容，提示确认。
- 提交成功后进入审核中状态页，并清除对应草稿。

内容规则：

- 不发布考试进行中的内容。
- 不发布未公开具体原题。
- 不包含个人敏感信息。
- 审核中内容只在“我的投稿”可见。

### 7.9 我的页面

页面目标：

- 让登录用户在电脑端集中管理收藏、投稿和账号设置。
- 未登录用户看到清晰登录入口，但仍能返回公开信息页。

未登录状态：

- 展示登录引导。
- 说明登录后可收藏学校、发布面经、查看投稿审核。
- 保留返回首页、院校、面经入口。

已登录状态：

1. 用户信息区
   - 昵称。
   - 年级。
   - 匿名偏好。

2. 我的收藏
   - 收藏学校。
   - 收藏面经。

3. 我的投稿
   - 审核中。
   - 已发布。
   - 被退回。

4. 站内提醒
   - 收藏学校的最近节点。
   - 投稿审核状态变化。

5. 账号设置
   - 修改昵称。
   - 修改年级。
   - 默认匿名展示。
   - 退出登录。

桌面规则：

- 收藏和投稿列表可用表格或紧凑卡片。
- 投稿状态必须清晰展示，不把审核中内容放到公共面经入口。
- 退出登录时清除本地面经草稿。

## 8. 管理端页面设计

### 8.1 管理端框架

页面目标：

- 让运营和审核人员以稳定桌面后台方式处理官方数据和社区内容。
- 保持队列、详情、来源、预览和操作记录在同一工作流内。

页面结构：

1. 左侧导航
   - 使用 `src/lib/product.js` 中的管理端工作流导航标签。
   - 当前导航项必须可见，不能只靠颜色表达。

2. 顶部状态栏
   - 页面标题。
   - 当前筛选状态或队列数量。
   - 当前管理员身份。
   - 退出登录。

3. 队列主区
   - 表格展示待处理项。
   - 支持按状态、学校、年份、更新时间、风险等级筛选。
   - 行点击后打开右侧详情面板。

4. 右侧详情面板
   - 结构化字段。
   - 来源或材料摘要。
   - 风险提示。
   - 学生端可见内容预览。
   - 操作按钮和原因填写。

通用规则：

- 所有管理端页面必须校验管理员角色。
- 未登录或无权限时不得展示队列数据、认证材料、举报人信息或内部审核记录。
- 所有发布、隐藏、删除、退回、限制账号操作必须填写或自动记录原因。
- 影响学生端可见内容的操作必须先展示学生端预览。
- 管理端可以看到必要的审核材料，但不得把原始材料、手机号、真实姓名、来源账号、私有材料 URL 渲染到学生端预览。

### 8.2 数据入库页

页面目标：

- 管理 AI 检索和字段抽取任务。
- 帮助运营找到官方来源并生成待审核草稿。

页面结构：

1. 创建任务区
   - 年份。
   - 学校。
   - 关键词。
   - 来源优先级提示。
   - 创建任务按钮。

2. 任务队列表格
   - 学校。
   - 年份。
   - 状态：运行中、已完成、失败、已审核。
   - 来源数量。
   - 抽取置信度。
   - 创建人。
   - 创建时间。

3. 任务详情面板
   - 来源候选列表。
   - 来源优先级。
   - 抽取字段。
   - 置信度。
   - 待人工确认项。
   - 生成简章草稿按钮。

规则：

- AI 抽取结果只能进入草稿，不能自动发布。
- 第三方来源只能标为线索，不能作为最终官方来源直接发布。
- 来源候选必须保留优先级和可追溯链接。
- 失败任务展示可重试入口，但不得暴露系统内部错误细节。

### 8.3 简章审核页

页面目标：

- 审核官方简章草稿并发布学生端可见版本。
- 让审核员同时核对结构化字段、官方来源和学生端预览。

页面结构：

1. 审核队列表格
   - 学校。
   - 年份。
   - 状态。
   - 来源类型。
   - 待补字段数。
   - 更新时间。

2. 审核详情面板
   - 左侧结构化字段。
   - 右侧官方来源预览或链接。
   - 字段级状态：已确认、待补充、有疑问。
   - 学生端预览。

3. 操作栏
   - 保存草稿。
   - 提交审核。
   - 发布。
   - 退回。
   - 归档。

规则：

- 发布前必须有官方来源链接。
- 关键字段缺失时可发布，但必须标记“官方未明确/待补充”。
- 发布后版本号递增并记录操作人和时间。
- 发布后学生端只能看到已发布版本，不得看到草稿或待审核内容。

### 8.4 时间线管理页

页面目标：

- 查看由简章生成的时间线节点。
- 必要时人工覆盖节点日期、标题或说明。

页面结构：

1. 筛选区
   - 学校。
   - 年份。
   - 节点类型。
   - 是否人工覆盖。

2. 节点表格
   - 学校。
   - 年份。
   - 节点类型。
   - 日期。
   - 日期精度。
   - 来源。
   - 是否人工覆盖。
   - 学生端状态。

3. 节点编辑面板
   - 原始自动生成值。
   - 人工覆盖值。
   - 覆盖原因。
   - 学生端预览。

规则：

- 日期未知时展示“待公布”。
- 人工覆盖必须填写原因。
- 覆盖后保留原始自动生成数据。
- 公开时间线 API 继续只输出已发布规则下允许展示的节点。

### 8.5 公式管理页

页面目标：

- 配置学校年份公式，保证学生端计算器准确可追溯。
- 发布前完成来源校验和测试样例验证。

页面结构：

1. 公式列表
   - 学校。
   - 年份。
   - 公式名称。
   - 发布状态。
   - 来源状态。
   - 样例测试状态。
   - 更新时间。

2. 公式编辑器
   - 学校。
   - 年份。
   - 公式名称。
   - 输入项名称。
   - 输入项满分。
   - 输入项是否必填。
   - 权重或自定义计算配置。
   - 公式说明。

3. 测试样例区
   - 样例输入。
   - 预期结果。
   - 实际结果。
   - 测试状态。

4. 学生端预览
   - 公式说明。
   - 输入项。
   - 分项贡献。
   - 免责声明。

规则：

- 没有来源链接不能发布。
- 发布前必须通过至少 1 个测试样例。
- 公式发布后学生端计算器才可选择。
- 学生端预览不得展示草稿内部备注。

### 8.6 面经审核页

页面目标：

- 审核学生提交的结构化面经。
- 保护学生隐私，拦截违规内容。

页面结构：

1. 待审队列

   - 阶段。

3. 操作栏
   - 通过。
   - 退回修改。
   - 隐藏。
   - 限制账号。

规则：

- 审核员不得在学生端展示手机号、真实姓名、认证材料、来源账号或私有材料 URL。
- 含有未公开具体原题的内容应退回改写为题型类别。
- 售卖资料、包过、代写、外部引流直接隐藏。
- 审核通过后才可注册为学生端可见面经。

### 8.7 认证审核页

页面目标：

- 审核可选认证材料，给面经增加可信标签。
- 保证认证材料只在后台审核需要范围内可见。

- 前台只展示认证标签，不展示材料。
- 审核拒绝时需填写原因。
- 学生端、匿名预览、我的页面不得展示原始材料、私有材料 URL 或可识别身份字段。

- 隐藏或删除内容后，学生端列表和详情必须同步不可见。

## 9. 核心组件规范

### 9.1 桌面表格

使用场景：

- 院校列表。
- 最新简章。
- 我的收藏学校。
- 管理端队列。

规则：

- 表头固定字段名称，避免只靠图标表达。
- 行内操作保持右对齐。
- 状态标签必须有文字。
- 长字段可以截断，但关键字段如学校、年份、截止时间不能同时被隐藏。
- 表格在 768-1023px 过渡宽度下可转为卡片。

### 9.2 筛选侧栏

使用场景：

- 院校列表。
- 时间线。
- 面经列表。
- 管理端队列。

规则：

- 常用筛选项直接展开。
- 当前筛选条件必须可见。
- 提供清空筛选按钮。
- URL 查询参数应反映当前筛选状态。

### 9.3 右侧信息栏

使用场景：

- 院校详情。
- 计算器。
- 面经详情。
- 发布面经。
- 管理端详情和审核面板。

规则：

- 只放目录、关键操作、来源摘要、提交摘要或个人状态。
- 不放长正文。
- 宽度不足时移动到主内容下方。
- 右侧栏中的主按钮不得与主内容同屏出现第二个主按钮。

管理端例外：

- 管理端右侧审核面板可以包含结构化字段、来源摘要、风险提示和学生端预览。
- 管理端右侧审核面板可以同时出现多个审核动作，但破坏性动作必须有确认或原因填写。

### 9.4 状态标签

通用状态：

- 已发布
- 待发布
- 待审核
- 已认证
- 历史参考
- 即将截止
- 已截止
- 待公布
- 待补充

规则：

- 使用浅底深字。
- 颜色不能作为唯一表达。
- 状态含义与移动端一致。

### 9.5 空状态

规则：

- 空状态必须给出下一步行动。
- 不使用大插画。
- 可使用小图标、短文案和一个按钮。

示例：

- “没有符合条件的院校，试试切换年份或状态。”
- “还没有收藏学校，收藏后这里会显示你的时间线。”
- “该校该年份暂无明确公开公式，暂不支持计算。”

### 9.6 管理端审核操作栏

使用场景：

- 简章审核。
- 公式发布。
- 面经审核。
- 认证审核。
- 举报处理。

规则：

- 主操作放在稳定位置，破坏性操作使用危险样式。
- 发布、隐藏、删除、退回、限制账号必须填写或自动记录原因。
- 操作后队列状态和详情面板同步更新。
- 操作失败时保留当前审核上下文和已填写原因。

### 9.7 学生端预览

使用场景：

- 简章审核。
- 公式管理。
- 面经审核。
- 认证审核。
- 举报处理。

规则：

- 预览内容必须等同或严格接近学生端可见内容。
- 预览不得展示审核员备注、原始认证材料、手机号、真实姓名、来源账号、私有用户 ID、私有材料 URL。
- 草稿或待审核内容必须明确标注不会立即公开。

## 10. 表单与交互规范

### 10.1 搜索与筛选

- 桌面端筛选优先使用左侧栏或顶部工具条。
- 筛选改变后保留当前位置和 URL 查询参数。
- 筛选结果为空时保留筛选条件，并提供清空筛选。
- 全局搜索可以跳转到院校或面经结果页，但不得混淆官方简章和社区内容来源。

### 10.2 登录

- 登录流程复用移动端手机号验证码逻辑。
- 登录页或弹层必须支持 `returnTo`。
- 由收藏、有用、举报触发登录时，登录成功后继续执行原操作。
- API 风格请求继续返回 JSON `login_required`，HTML 请求使用共享 `/login` 流程。
- 管理端登录后必须校验角色；无管理员角色时展示无权限页面，不展示后台数据。

### 10.3 表单

- 必填字段使用清晰标记。
- 错误提示靠近对应字段。
- 长文本提供字数提示。
- 提交失败时保留已填写内容。
- 认证材料上传前必须说明用途和可见范围。

### 10.4 反馈

- 收藏、点有用、举报成功后使用轻量 toast。
- 重要错误必须在页面内有非 toast 文案兜底。
- 发布面经成功后进入审核中状态页。
- 计算器输入错误必须说明下一步，例如“请输入 0-750 之间的高考成绩”。
- 管理端审核操作成功后更新队列状态，并在详情面板保留操作结果。

## 11. 可访问性与适配

### 11.1 可访问性

- 所有图标按钮必须有可访问名称。
- 表格行内操作必须可键盘聚焦。
- 焦点态必须可见。
- 表单 label 与输入控件明确关联。
- 状态不能只用颜色表达。
- 重要错误必须有文字说明。

### 11.2 响应式行为

- 学生端 1280px 及以上：优先使用完整桌面布局。
- 学生端 1024-1279px：保留顶部导航和主要双栏，右侧信息栏可下移或变窄。
- 学生端 768-1023px：列表筛选可收敛到顶部区域，详情页收敛为单栏加目录。
- 学生端低于 768px：遵守移动端 PRD。
- 管理端 1280px 及以上：使用左侧导航、队列表格、右侧详情面板。
- 管理端 1024-1279px：保留左侧导航和队列表格，详情面板可改为抽屉。
- 管理端低于 1024px：不作为核心验收宽度，可提示使用电脑完成审核。

### 11.3 布局质量

- 不出现横向滚动。
- 文本不得互相重叠。
- 长学校名称、长专业名称、长来源标题必须能换行或截断。
- 固定顶部导航不得遮挡锚点跳转后的标题。
- 粘性右侧栏不得遮挡页脚或主要操作。
- 管理端表格和详情面板在 1280px、1440px、1920px 宽度下不重叠。

## 12. 前端验收标准

### 12.1 学生端桌面验收

- 顶部导航包含首页、院校、时间线、计算器、面经、我的。
- 首页首屏展示核心任务入口、最近节点、最新简章或面经概览，不变成营销页。
- 院校列表支持桌面筛选、排序、搜索和结果数量展示。
- 院校详情页右侧能进入官方简章、报名入口、计算器和面经，同时遵守单主按钮规则。
- 时间线页支持全部节点和我的收藏切换，未登录进入统一登录流程。
- 计算器无明确公式时不展示输入和计算按钮。
- 面经投稿审核前不在公共列表展示。
- 匿名面经不暴露账号和认证材料。

### 12.2 管理端桌面验收

- 管理端使用左侧导航、顶部状态栏、队列表格和右侧详情面板或抽屉。
- 数据入库结果只能生成草稿，不能自动发布。
- 简章发布前能看到官方来源、字段状态和学生端预览。
- 时间线人工覆盖必须填写原因，并保留原始自动生成数据。
- 公式发布前必须填写来源链接并通过至少 1 个测试样例。
- 面经审核能看到敏感内容提示和学生端预览。
- 认证材料只在后台审核页面可见，学生端预览只展示认证标签。
- 举报处理能记录操作人、处理时间和处理原因。
- 无管理员角色访问管理端时不得展示后台队列、认证材料或举报信息。

### 12.3 视觉验收

- 电脑端沿用蓝绿清爽、白底浅灰背景、克制卡片或面板风格。
- 卡片圆角不超过 8px。
- 不使用大面积渐变、复杂插画、营销式 hero。
- 宽屏下内容不无限拉伸，正文行宽可读。
- 学生端表格、筛选侧栏、右侧信息栏在 1024px、1280px、1440px、1920px 宽度下不重叠。
- 管理端左侧导航、队列表格、右侧详情面板在 1280px、1440px、1920px 宽度下不重叠。

### 12.4 交互验收

- 筛选状态保留在 URL 查询参数中。
- 无 JavaScript 时核心 GET 查询和详情页可用。
- 登录后能回到触发来源或继续执行收藏、有用、举报等 pending action。
- 表单离开前提示未保存内容。
- 草稿保存和清除规则与移动端一致。
- 管理端审核操作失败时保留当前上下文和已填写原因。

### 12.5 产品边界验收

- 页面不得出现录取概率、排名预测、保证录取、内部消息、付费咨询、开放评论、私信等能力或文案。
- 官方来源不得把第三方整理内容标为官方。
- 计算结果必须保留“不代表录取概率”免责声明。
- 公开页面和匿名输出不得泄露手机号、真实姓名、认证材料、来源账号、私有用户 ID、私有材料 URL。
- 管理端学生端预览不得泄露后台备注、原始认证材料、举报人私密信息或私有存储路径。

## 13. 与现有前端 PRD 的关系

`docs/frontend-prd.md` 是移动优先前端 PRD，并已经定义学生端移动布局、管理端桌面后台、核心组件、文案、可访问性和验收要求。本电脑 Web 版 PRD 是学生端桌面体验和管理端桌面体验的补充：

- 业务逻辑以 `docs/prd.md` 为准。
- 移动端体验以 `docs/frontend-prd.md` 为准。
- 电脑端学生体验以本文档第 7 章为准。
- 电脑端管理体验以本文档第 8 章和 `docs/frontend-prd.md` 第 6 章共同约束。
- MVP 不单独定义手机版管理端；管理端核心验收以电脑 Web 为准。
- 如果三份文档出现冲突，优先级为：产品边界和数据权限遵守 `docs/prd.md`，端侧布局遵守对应前端 PRD。
